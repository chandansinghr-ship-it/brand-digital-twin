# [P1.2] Observability — Durable Error Sink + Alert Rules

> Grounded in upstream `brand-digital-twin` @ `fb03ddd`. `observability.ts` has a
> `MetricsTracker` (spans, `recordMetric`, `raiseAlert`, `getAverageLatency`) and a
> `PinoLogger`, plus the `/ready` DB-ping (landed `fb03ddd`). The gap: **everything
> is in-memory** — metrics, spans, and alerts live in `MetricsTracker` arrays, so
> they vanish on restart, aren't queryable across instances, and `raiseAlert()`
> isn't wired to any threshold. Blind in prod.

---

## P1.2a — Durable, tenant-scoped error sink

### Build
1. **`error_events` table** (Supabase/Postgres, via the P1.4 migration system):
   ```sql
   CREATE TABLE IF NOT EXISTS brand_twin.error_events(
     event_id    TEXT PRIMARY KEY,
     tenant_id   TEXT,                  -- nullable: pre-auth errors
     severity    TEXT NOT NULL,         -- 'error' | 'warning' | 'critical'
     source      TEXT NOT NULL,         -- module/operation
     message     TEXT NOT NULL,
     context     JSONB,                 -- REDACTED — never tokens/PAN/secrets
     trace_id    TEXT,                  -- correlate to MetricsTracker span
     created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
   );
   ```

2. **`recordError(evt)` on the sink** — writes a row; **token-redaction is
   mandatory and centralized** (one allowlist-based scrubber over `context` before
   write — strip anything matching bearer/refresh/`Authorization`/`*_secret`/
   `*_key`/PAN patterns). This is the same guarantee P1.6 audits for.

3. **Swappable webhook:** an `ErrorSink` interface with two implementations —
   the DB sink (default) and an optional Sentry-compatible HTTP forwarder. Config
   selects; **no hard vendor lock**. Both receive the *already-redacted* event.

4. **Wire it in:** `MetricsTracker.endSpan(..., 'failure', error)` and the
   `server.ts` top-level error handler call `recordError`, carrying the span's
   `traceId` so a persisted error joins its timing trace.

### Build checklist
- [ ] `error_events` table (as a P1.4 migration)
- [ ] Central redaction scrubber + unit tests proving no token survives
- [ ] `ErrorSink` interface; DB impl + optional webhook impl
- [ ] `recordError` called from span failures + server error handler, with `trace_id`
- [ ] Tests: a forced failure persists a redacted, tenant-scoped row

### Done when
- A forced error appears in `error_events` with a tenant-scoped trace and **no
  token in the payload** (the PROD_READINESS_PLAN P1 exit-gate line).

## P1.2b — Metrics persistence + alert rules

### Problem
`recordMetric` accumulates `Metric[]` in memory; `raiseAlert` just pushes a string.
No thresholds fire automatically, and metrics don't survive a restart or aggregate
across instances.

### Build
1. **Metric durability/scrape:** either flush `MetricsTracker` metrics to a
   `metrics` table periodically, or (preferred for ops) expose a
   **`GET /metrics` Prometheus endpoint** so an external scraper owns retention.
   Pick one; the scrape endpoint is lower-maintenance and multi-instance-correct.
2. **Alert rules** — a small evaluator (runs on a timer or per-scrape) over the
   live signals, calling `raiseAlert()` + `recordError(severity:'critical')` when
   a rule trips:
   | Rule | Threshold (tune) | Why |
   |------|------------------|-----|
   | Job-queue backlog | pending jobs with `run_at` past due > N for > M min | scheduler stalled |
   | Adapter error rate | failure spans / total > X% over window | provider outage / bad token |
   | POAS-calc duration | p95 > T ms | perf regression |
   | Backup staleness | newest backup older than cadence + grace | P1.4b hook |
   | `/ready` failing | DB or queue unreachable | hard outage |
3. **Alert delivery:** route `raiseAlert` to a real channel (webhook/email/Slack)
   behind the same swappable interface as the error webhook.

### Build checklist
- [ ] Metric durability: `GET /metrics` (Prometheus) **or** periodic flush table
- [ ] Alert-rule evaluator with the 5 rules above (thresholds in config)
- [ ] `raiseAlert` → real delivery channel (swappable)
- [ ] Tests: each rule trips on synthetic data; recovers when signal clears

### Done when
- A simulated job backlog and a simulated adapter-error spike each fire an alert
  to the configured channel, and metrics survive a process restart.

## Sequencing
P1.2a (sink) first — the error path is the riskiest blind spot and P1.2b's alert
evaluator reuses `recordError`. Both depend on P1.4a migrations for their tables.
Land before P1.7 so the load test has instrumentation to read.
